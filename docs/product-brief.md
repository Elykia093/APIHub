# APIHub · 产品简报

## 问题与目标

公益站用户通常分散维护多个不同实现的 API 站点，需要每天逐个打开页面签到，也容易错过停机、迁移、活动和规则调整公告。

核心 JTBD：作为公益站使用者，我希望在一个自托管页面中安全地维护站点访问令牌，自动完成可自动化的每日签到，并集中查看新公告，从而减少重复操作和信息遗漏。

## 默认用户与约束

- 单管理员、自托管、单应用实例，PostgreSQL 作为唯一持久存储。
- 首批适配器覆盖 New API、Sub2API 与 ZenAPI，并按能力决定是否支持签到或公告。
- 使用站点生成的访问令牌和用户 ID，不保存用户名、密码或浏览器 Cookie。
- 默认仅允许访问公网 HTTPS 站点；私网和 HTTP 必须显式开启。
- 遇到 Turnstile、验证码、二次验证或站点拒绝自动化时，任务记录为“需人工处理”，不尝试绕过。

## MVP 范围

1. 站点管理：新增、查看、启用/停用站点，配置签到和公告同步计划。
2. 凭据保护：访问令牌使用独立主密钥加密保存，API 永不回显明文。
3. 签到：支持手动触发和定时执行；同一站点同一自然日幂等；保存奖励、结果和错误分类。
4. 公告：同步 `/api/status` 的结构化公告和 `/api/notice` 的文本通知；按内容指纹去重。
5. 管理页：查看站点状态、最近签到结果、公告流，并可手动触发任务。
6. 运维基线：健康检查、结构化日志、请求 ID、外部请求超时、Docker Compose 部署。

## 明确不做

- 不做代理网关、模型路由、余额管理、计费或 API Key 聚合。
- 不做用户名密码登录自动化、OAuth 托管、浏览器 Cookie 捕获、页面点击签到、验证码识别或风控绕过。
- 不做多用户、租户、RBAC、移动 App、浏览器插件。
- 首版不做邮件、Telegram、Bark 等外部通知渠道。
- 首版不承诺多实例调度；生产默认保持一个副本。

## 验收标准

- Given 适配器声明支持签到且凭据合法，When 手动或计划任务触发签到，Then 系统创建一条可追溯记录并返回成功、今日已签到、需人工处理或失败之一。
- Given 选择自动识别，When 站点存在可验证的适配器信号，Then 保存具体适配器及其能力；无法识别时拒绝创建而不是猜测。
- Given 同一站点同一天重复触发，When 前一次已成功或已签到，Then 不再次调用上游签到接口。
- Given 上游超时、非 JSON、HTTP 失败或业务失败，When 任务结束，Then API 不返回伪成功，记录稳定错误分类且日志不含令牌。
- Given `/api/status` 或 `/api/notice` 返回重复内容，When 多次同步，Then 公告表只保留一个内容指纹并更新最后发现时间。
- Given 未携带或携带错误管理员令牌，When 请求管理 API，Then 返回 401 且不泄露资源是否存在。
- Given API 返回站点信息，Then 访问令牌字段只显示是否已配置，不回显密文或明文。

## 指标与停止条件

- 核心指标：启用站点的每日签到完成率。
- 护栏指标：重复签到调用数、需人工处理比例、上游超时率、公告重复写入数、凭据泄露事件数。
- 暂停或终止条件：站点停用、主密钥不可用或数据库不可写。明确需要验证码时，当天签到记录进入终态、不再重试；次日仍按计划再次尝试。

## 后续版本候选

- 通知渠道和公告关键词过滤。
- One API、OneHub、DoneHub 等独立适配器。
- 凭据轮换、导入导出与备份恢复。
- 多实例任务租约、审计事件和细粒度权限。
